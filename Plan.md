# LoveMemory Upgrade Plan

## 当前项目理解

- 这是一个基于 Vue 3、TypeScript、Vite、Pinia、Vue Router 和 Tailwind CSS 4 的多页面恋爱纪念网站。
- 当前已经具备首页、故事页、相册页、旅程页、照片页和音乐页的基础路由结构，并使用 `src/data/*` + `src/stores/couple.ts` 管理核心展示数据。
- 现有项目的优点是：
  - 页面主题明确，已经围绕“恋爱纪念网站”展开
  - 有樱花、鱼群、水面、时间线、地图等具有记忆点的视觉元素
  - 目录结构大体清晰，页面与组件拆分有基础
- 现有项目的主要不足是：
  - 首页承载的信息较少，不能很好地建立整体叙事和仪式感
  - `LovePhotoPage`、`MusicPage` 仍是占位页
  - 页面气质尚未完全统一，部分模块更像 demo 而不是完整成品
  - 移动端导航和部分模块适配不够完整
  - 纪念文案、照片、旅程、歌单等内容虽然有数据基础，但可维护性和扩展性还可以更进一步

## 本轮升级目标

1. 保留原有页面骨架与主题，不做无意义重写。
2. 提升整体视觉质感，方向为：温柔、浪漫、简洁、有仪式感。
3. 增强首页叙事能力，让它成为纪念站入口，而不是单纯展示标题。
4. 完善故事、照片墙、旅程、歌单等模块，让网站更完整。
5. 优化移动端导航、页面结构层次和浏览体验。
6. 将可替换内容集中在 `src/data/*`，降低后续维护成本。

## 实施方案

### 1. 主题与全局体验升级

- 重构 `src/assets/main.css` 主题变量，统一颜色、背景层次和卡片质感。
- 优化全局布局与页脚，让整站视觉语言保持一致。
- 增强移动端导航与页面切换体验。
- 为装饰性动画增加 reduced motion 兼容。

### 2. 首页升级

- 保留原有“标题 + 纪念时长 + 数据统计”主骨架。
- 增加更完整的 Hero 文案、纪念说明、纪念节点和章节入口。
- 让首页承担“纪念馆首页”的叙事角色。

### 3. 内容页补全

- 故事页：增强时间线的章节感、地点感和情绪表达。
- 相册页：保留原有特色轮播，同时增加更实用的照片墙。
- 照片墙页：从占位页升级为独立浏览页。
- 旅程页：保留地图，同时新增旅程摘要和城市卡片。
- 歌单页：从占位页升级为可维护的情绪歌单页。

### 4. 数据可维护性升级

- 扩展 `src/types/index.ts` 中的业务类型。
- 将首页文案、纪念节点、故事字段、照片扩展信息、歌单信息集中在 `src/data/*`。
- 继续复用 `Pinia store` 作为页面数据统一入口。

## 本轮输出结果预期

- 网站不再是“半成品演示”，而是一版可以继续打磨和持续填充内容的恋爱纪念站。
- 后续替换文字、图片、日期时，优先在 `src/data/couple.ts`、`src/data/story.ts`、`src/data/gallery.ts`、`src/data/journey.ts`、`src/data/playlist.ts` 中维护。

## 本轮性能优化补充

- 目标：在不改现有页面结构的前提下，缩短首页首加载时间，优先优化首屏必须下载、解析和执行的资源。
- 已确认的问题：
  - 首页共享 `couple store` 里提前引入了故事、相册、旅程、歌单全部数据，放大了首屏共享 JS chunk。
  - 根布局首屏就挂载了樱花和鱼群装饰，增加了非必要的首屏脚本执行与动画开销。
  - `index.html` 同步加载 `sakura.min.js`，会阻塞首屏脚本链路。
  - 全局背景图 `src/assets/background.jpg` 体积偏大，且使用了 `background-attachment: fixed`。
  - Google Fonts 通过 CSS `@import` 进入关键渲染路径，会额外增加字体请求等待。
- 本轮处理方向：
  - 首页共享 store 只保留首页和全局布局必需的 `coupleData`。
  - 各内容页改为就近引入各自的数据源，恢复路由级数据拆分。
  - 樱花脚本改为组件内异步加载，移出 `index.html` 首屏阻塞链路。
  - 装饰组件改为异步组件，并延后到首屏渲染之后或进入视口后再挂载。
  - 背景图改用更轻量的 WebP 资源，并移除固定背景附着带来的额外渲染成本。
  - Google Fonts 从 CSS `@import` 调整为 HTML 中的预连接和非阻塞加载。

## 本轮图片替换补充

- 目标：保持现有页面结构不变，将项目中 `picsum.photos` 样例图替换为 `src/assets` 中提供的线条小狗本地图片。
- 已确认的替换范围：
  - `src/data/gallery.ts` 中 8 张相册样例图。
  - `src/data/story.ts` 中 4 张故事时间线样例图。
  - `src/data/journey.ts` 中 12 张旅程城市卡片样例图。
- 本轮处理方向：
  - 新增统一的本地图片映射文件，避免在多个数据文件中重复书写图片路径。
  - 按现有模块展示形式分配图片比例：
    - 相册优先分配横图、竖图、方图给对应的 `wide`、`tall`、`square` 卡片。
    - 故事页优先使用更适合横幅裁切的横向图片。
    - 旅程页使用其余本地图片替换卡片示例图。
  - 保持 `src/data/*` 作为图片可维护入口，后续替换时只需调整映射文件和对应数据源。

## 本轮 Footer 鱼群修正补充

- 目标：保持现有结构不变，以当前笔记本屏幕上的观感为基准，修正 footer 鱼群在手机和更大屏幕下“过小 / 过慢”与“过大 / 迟缓”的问题。
- 已确认的问题：
  - 鱼体尺寸当前基本按容器宽度线性缩放，导致手机端明显偏小，大屏明显偏大。
  - 水平速度当前也按容器宽度线性变化，虽然跨屏时间比例接近，但体感并不稳定。
  - 大屏下鱼数量还会按面积继续增加，进一步放大“画面迟缓”的观感。
- 本轮处理方向：
  - 用笔记本宽高作为参考尺寸，分别对鱼体缩放和运动速度做上下限钳制。
  - 将动画更新改为按帧时间步进，降低不同设备帧率波动带来的迟滞感。
  - 收敛鱼的数量增长范围，让手机和大屏都更接近当前满意的视觉密度。
